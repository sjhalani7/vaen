Portable agent harness sharing

Share agent setups without manual file packing.

An agent setup is the instruction files, reusable skills, and optional tool config that tell a coding assistant how to behave in a repo.

Today, sharing one usually means manually collecting those files, then hoping another person unzips them into the right places.

VAEN replaces that with a manifest and a CLI flow: you write agent.yaml, VAEN builds the single shareable .agent archive with the declared setup files and metadata, and the receiver imports it with one command.

What VAEN does

Turns an agent setup into a portable archive.

An agent.yaml file is the recipe for the bundle. It tells VAEN which instruction file is the main entry point, which optional files and skill directories belong with it, which MCP servers should be declared, and which credential variable names the receiver needs to provide locally.

Why it exists

  • There is no common package format for agentic coding harnesses.
  • Manual zip-and-copy sharing is easy to get wrong.
  • Receivers should not need to guess where every file belongs.
  • Teams need a repeatable way to inspect and import setups.

What agent.yaml does

  • Names the files and directories that make up the harness.
  • Separates credential variable names from credential values.
  • Gives VAEN enough structure to validate, build, and inspect.
  • Lets imports recreate the setup in the right target locations.

Included in bundles

  • Main instructions and optional instruction includes
  • Skill directories and their files
  • Project-scoped MCP declarations
  • Metadata used for import, inspection, and validation

Keep out of bundles

  • Builders should exclude credential values, .env files, and private keys
  • Builders should leave out OAuth state, keychains, and local secret stores
  • MCP server implementations are referenced, not bundled
  • Dependency caches and runtime environments stay outside the bundle

Workflow

Validate, build, inspect, import, verify, and clean up.

The manifest is the source of truth. VAEN checks the manifest, builds a portable archive, shows what is inside, imports it into the target repository, and verifies the materialized structure.

  1. 01

    Validate

    Check required fields, supported artifact types, paths, and declaration structure.

    vaen validate -f agent.yaml
  2. 02

    Build

    Create an OCI-style archive. If -o is omitted, VAEN writes a default .agent name.

    vaen build -f agent.yaml -o team-setup.agent
  3. 03

    Inspect

    Print bundle metadata, normalized paths, instructions, skills, MCP declarations, and required variable names.

    vaen inspect team-setup.agent
  4. 04

    Import

    Write the canonical stored copy first, then activate files for the target repository and selected client.

    vaen import team-setup.agent --client codex
  5. 05

    Doctor

    Report required names and check structure, placement, and local config syntax.

    vaen doctor --client codex
  6. 06

    Cleanup

    Remove only the canonical stored copy after activated files have been reviewed.

    vaen cleanup team-setup.agent

Install

Install from GitHub or a local clone.

The CLI provides validate, build, inspect, import, doctor, and cleanup.

MCP examples

Use --client when importing bundles with MCP declarations.

MCP support is host-neutral in the manifest. During import, VAEN writes the selected client format for Codex, Claude Code, or Copilot. Bundles that contain MCP declarations require --client on import.

Archive and import layout

Stable bundle paths become materialized repository paths.

Builders can reference source paths from inside or outside a repo, including paths such as ~/.codex/AGENTS.md. VAEN stores those files at stable bundle paths and materializes them into the target repository during import.

Activated outputs

Default import for instructions/skills bundles
Writes root instructions to AGENTS.md and CLAUDE.md, and mirrors skills to .agent/skills/... and .claude/skills/....
--client claude
Defaults activated outputs to CLAUDE.md and .claude/skills/....
--client codex
Defaults activated outputs to AGENTS.md and .codex/skills/....
--client copilot
Defaults activated outputs to AGENTS.md and .copilot/skills/....
MCP client config
Written to .codex/config.toml, .mcp.json, or .github/mcp.json when --client is used.

Safety model

VAEN is designed to move setup, not secrets.

Manifest authors must write credential variable names such as OPENAI_API_KEY, not credential values. Secrets do not belong in manifests, instruction files, skills, references, assets, or any other bundled file.

Overwrite protection

  • Existing root instruction files block import.
  • Existing .agent/<bundle-name> directories block import.
  • Existing mirrored skill names block import.
  • Existing MCP client config files block import.

Credential handling

  • requiredVars and MCP env fields store variable names only.
  • http_headers is for non-secret static HTTP headers.
  • Build rejects obvious declared secret file paths such as .env, *.pem, *.key, and id_rsa.
  • Rejected secret-like paths are reported by path only; VAEN does not print file contents or credential values.

Doctor scope

  • Reports required environment variable names as warnings.
  • Checks imported structure, expected file placement, and local config syntax.
  • Does not read .env files, keychains, or credential stores.
  • Does not connect to MCP servers or validate auth/API credentials.

FAQ

Details that matter during packaging and import.

What does vaen cleanup delete?

Cleanup deletes .agent/<bundle-name> only. It does not delete root instruction files, skill mirrors, or MCP client config.

Can a manifest contain API keys or token values?

No. Manifests store credential variable names only, such as DOCS_MCP_TOKEN. Do not put secret values, NAME=value assignments, password-bearing URLs, or credential files in manifests or bundled files.

Does doctor verify authentication?

No. doctor reports required variable names and checks structure, expected file placement, and local config syntax. It does not validate remote APIs, MCP server auth, or credential values.

What happens if an output file already exists?

VAEN fails before overwriting existing setup files, including root instruction files, stored bundle directories, mirrored skill names, and MCP client config files.

Where do included instruction files go?

On import, VAEN creates root instruction file(s) from instructions.main when the target files do not already exist. Default import writes both AGENTS.md and CLAUDE.md; --client or target overrides can select one output. Extra files listed in instructions.includes stay in .agent/<bundle-name>/instructions/...; they are not concatenated into the root instruction file(s).
